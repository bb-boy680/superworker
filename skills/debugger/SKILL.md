# Debugger Skill

三步调试法：**埋点 → 分析 → 修复 → 清理**

## ⚠️ 强制性流程警告（Step Lock）

**本 skill 必须严格按照以下步骤顺序执行，严禁跳步！**

### 步骤锁机制

```
┌─────────────────────────────────────────────────────────────┐
│  前置步骤 → 步骤1 → 步骤2 → 步骤3                            │
│  (环境准备)  (埋点)   (分析)   (清理)                         │
│       ↓      ↓      ↓      ↓                                 │
│    [必须]  [必须]  [必须]  [必须]                              │
│       ↓      ↓      ↓      ↓                                 │
│   未完成   未埋点   未分析   未修复                            │
│   不可      不可    不可    不可                               │
│   进入      分析    修复    清理                               │
└─────────────────────────────────────────────────────────────┘
```

### 跳步的严重后果

**如果跳过埋点直接修复：**
- ❌ **修复无效**：没有数据支撑，修复可能是错误的
- ❌ **引入新 bug**：不了解完整执行流程，修复可能破坏其他功能
- ❌ **问题复发**：未定位根因，表面修复后问题会再次出现

**如果跳过分析直接修复：**
- ❌ **治标不治本**：只修复表象，根本原因未解决
- ❌ **连锁反应**：一处改动引发多个模块故障
- ❌ **调试困难**：后续排查更复杂，需要重复工作

### 强制执行规则

1. **步骤1 未完成（未添加埋点）→ 禁止进入步骤2**
   - 如果用户要求直接修复，必须拒绝并解释必须先埋点

2. **未运行代码收集日志 → 禁止进行分析**
   - 没有日志数据就无法分析，必须等待日志生成

3. **未分析日志定位根因 → 禁止进行修复**
   - 严禁凭经验猜测问题原因
   - 必须有日志数据支撑修复决策

4. **未获得用户确认 → 禁止清理埋点**
   - 修复必须验证成功后才能清理

## 前置步骤

```bash
# 设置环境变量并启动服务
export DEBUG_SESSION_ID=$(grep DEBUG_SESSION_ID .worker/.env | cut -d'=' -f2)
```

## 步骤 1：添加埋点

**目标**：在关键位置插入日志代码。

**开始前 Checklist**:
- [ ] 前置步骤已完成（环境变量设置 + 日志服务启动）
- [ ] 已识别需要调试的文件和关键位置
- [ ] 已确认项目类型（frontend/backend）

### 1.1 读取配置

读取 `.worker/debug/config.yaml`：
- `port`：日志服务端口
- `frontend`/`backend`：项目类型匹配规则

**Session ID**：从前置步骤设置的环境变量 `DEBUG_SESSION_ID` 获取

```javascript
const sessionId = process.env.DEBUG_SESSION_ID;
```

### 1.2 判断项目类型

根据文件路径匹配 `frontend` 或 `backend` 模式。

### 1.3 插入埋点代码块

**⚠️ 强制规则 - 必须严格遵守**

每个埋点位置必须是一个**完全独立的折叠块**，包含完整的、自包含的代码。**严禁**以下做法：

- ❌ **禁止**创建辅助函数（如 `__debugLog`）并在多处调用
- ❌ **禁止**引用外部模块或文件（如 `import('./debug-logger')`）
- ❌ **禁止**将埋点代码分散到多处或依赖外部状态
- ❌ **禁止**在文件顶部放置共享的埋点工具函数

**为什么这个规则是强制的：**

如果不遵守此规则，在步骤 3 清理埋点时会导致**严重的代码丢失意外**。清理脚本使用 `sed -i '/#region DEBUG/,/#endregion DEBUG/d'` 删除从 `#region DEBUG` 到 `#endregion DEBUG` 之间的**所有内容**。如果埋点代码依赖外部函数，这些外部函数会被错误地保留；如果埋点块内部调用了外部定义的函数，当埋点块被删除时，对外部函数的调用会丢失，但外部函数本身可能因在其他地方使用而无法删除，导致残留死代码。

**正确做法**：每个埋点块必须是独立的、完整的、自包含的代码单元。

在需要埋点的**每个关键位置**插入折叠块：

**前端**（JS/Vue/React）：
```javascript
// #region DEBUG [sessionId: xxx, port: yyy]
fetch('http://localhost:{{port}}/debug/log?session_id={{sessionId}}', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    location: '文件名.js:10',
    message: '[函数入口:进入函数]',
    data: { arg: value }
  })
}).catch(() => {});
// #endregion DEBUG
```

**后端**（Python）：
```python
# #region DEBUG [sessionId: xxx]
import json, time
with open(f'.worker/debug/logs/{{sessionId}}.log', 'a', encoding='utf-8') as f:
    f.write(json.dumps({
        'location': f'{__file__}:10',
        'message': '[函数入口:进入函数]',
        'data': {'arg': value},
        'timestamp': time.time()
    }, ensure_ascii=False) + '\n')
# #endregion DEBUG
```

**后端**（Node.js/TypeScript）：
```typescript
// #region DEBUG [sessionId: xxx]
import('fs').then((fs) => {
  const log = {
    location: `${__filename}:10`,
    message: '[函数入口:进入函数]',
    data: { arg: value },
    timestamp: Date.now()
  };
  fs.appendFileSync('.worker/debug/logs/{{sessionId}}.log', JSON.stringify(log) + '\n');
}).catch(() => {});
// #endregion DEBUG
```

> 详细代码示例参考：`references/implementation.md`

**步骤 1 完成 Checklist**:
- [ ] 已在所有关键位置插入埋点代码
- [ ] 每个埋点都是独立自包含的折叠块
- [ ] 没有创建外部依赖的辅助函数
- [ ] `#region DEBUG` 和 `#endregion DEBUG` 标记正确

## 步骤 2：日志分析 & 代码修复

**目标**：运行代码，分析日志，定位并修复问题。

### 🚨 步骤2 强制入口检查（Entry Gate）

**进入步骤2 必须满足以下条件，否则必须返回步骤1：**

- [ ] ✅ 步骤1 已完成（埋点已添加到代码中）
- [ ] ✅ **用户已明确确认复现了 BUG**（如回复"已复现"、"操作完成"）
- [ ] ✅ 日志文件已生成且有实际数据

**如果上述任一条件不满足，必须立即停止并等待用户！**

**特别注意**：
- **严禁自动轮询等待日志** - 必须等用户手动操作并确认
- **严禁假设用户已复现** - 必须收到用户明确回复
- **前端调试必须由用户手动触发** - 你无法代替用户点击按钮、输入内容

### 🚫 严禁跳步的严重后果

**违反以下规则将导致调试失败或引入新bug：**

| 违规行为 | 后果 | 风险等级 |
|---------|------|---------|
| 未运行代码就分析 | 无数据支撑，分析纯属猜测 | 🔴 极高 |
| 未看日志就修复 | 治标不治本，问题反复出现 | 🔴 极高 |
| 凭经验直接改代码 | 破坏其他功能，引入连锁bug | 🔴 极高 |

**如果不遵守此流程，可能导致：**
- 修复无效，浪费开发时间
- 引入新bug，影响线上功能
- 代码混乱，后续维护困难
- 问题复发，用户体验受损

### 步骤2 正确流程

```
步骤2.1 添加埋点完成
      ↓
🛑 等待用户手动复现 BUG
      ↓
用户确认"已复现"或"操作完成"
      ↓
步骤2.2 分析日志 → 定位根因
      ↓
步骤2.3 修复代码
      ↓
验证修复成功 → 进入步骤3
```

**⚠️ 再次强调**：
1. **埋点完成后必须等待用户** - 明确告知用户"请手动复现 BUG"
2. **收到用户确认后才能分析** - 严禁自动进行
3. **没有日志数据支撑，严禁进行任何修复操作！**

### 2.1 运行代码 & 等待用户复现

**⚠️ 关键：必须由用户手动复现 BUG，不能自动进行！**

**操作流程**：

```
┌─────────────────────────────────────────────────────────────┐
│  1. 告知用户："埋点已添加，请手动复现 BUG"                     │
│  2. 等待用户明确回复："已复现" 或 "操作完成"                   │
│  3. 确认日志文件生成后，才能进入 2.2 分析阶段                  │
└─────────────────────────────────────────────────────────────┘
```

**必须执行的操作**：
1. **明确提示用户**："埋点已完成，请手动操作复现 BUG（如：点击按钮、触发事件等）"
2. **等待用户确认**：必须收到用户明确回复（如"已复现"、"操作完成"、"日志生成了"）
3. **检查日志文件**：确认 `.worker/debug/logs/{sessionId}.log` 已生成且有内容

**🚫 严禁**：
- 禁止自动轮询等待日志文件
- 禁止假设用户已操作
- 禁止未获用户确认就进入分析

**用户确认后检查**:
- [ ] 日志文件 `.worker/debug/logs/{sessionId}.log` 已生成
- [ ] 日志中有实际数据记录

---

### 2.2 分析日志

**执行 Checklist**:
- [ ] 已读取 `.worker/debug/logs/{sessionId}.log` 日志文件
- [ ] 日志中有预期的埋点输出
- [ ] 已按时间顺序整理执行路径

读取日志文件：

```json
{"location":"user.js:42","message":"[变量检查:user对象]","data":{"id":null},"timestamp":1712345678901}
```

**分析要点**（必须回答以下问题）：
- 执行路径是否符合预期？（通过 `message` 中的范围判断）
- 变量值是否正确？（查看 `data` 字段）
- 边界条件处理是否有问题？
- 异步操作时序是否正常？

**⚠️ 如果日志数据不足以回答上述问题 → 回到步骤1添加更多埋点！**

### 2.3 修复代码

**🚨 强制规则：必须先完成日志分析，确认找到根本原因后才能修复！**

**修复前必须完成**:
- [ ] 已通过日志分析定位问题根因
- [ ] 能在日志中找到问题发生的证据
- [ ] 理解问题发生的完整原因
- [ ] 已确定修复方案（最小改动原则）

**修复原则**：
- 最小改动原则
- 先理解原因再修复
- 修复后验证

**修复后 Checklist**:
- [ ] 修复代码已保存
- [ ] 修复后运行测试/验证
- [ ] 问题已解决

**如果修复无效**:
- 回到步骤1添加更多埋点
- 重新运行收集更多日志
- 再次分析定位根因

---

## 步骤 3：移除埋点（需用户确认）

**目标**：清理所有调试埋点，恢复代码整洁。

**⚠️ 必须等待用户确认修复成功后再执行！**

### 3.1 清理埋点代码块

删除所有 `#region DEBUG` 到 `#endregion DEBUG` 之间的代码块：

```bash
# 删除多行（包括 #region 和 #endregion 行）
sed -i '/#region DEBUG/,/#endregion DEBUG/d' 文件名
```

### 3.2 清理原则

- 只删除 `[DEBUG]` 埋点
- 保留原有日志代码
- 如有问题，回到步骤 1

**步骤 3 完成 Checklist**:
- [ ] 所有 `#region DEBUG` 到 `#endregion DEBUG` 的代码块已删除
- [ ] 代码恢复正常状态（无残留埋点）
- [ ] 清理后代码运行正常
