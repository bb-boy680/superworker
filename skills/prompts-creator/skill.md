# Prompts Creator

## Overview

创建高质量提示词和 Claude Skill 的方法论指南。

**核心方法论**：三层防御体系 + 六大原则

**支持**：创建 / 优化 / 质量检测

---

## 一、核心公式

```
提示词效果 = 信任模型 + 清晰结构 + 渐进式提示

Skill 质量    = 解释原因 × 正面指令 + description兜底
普通提示词质量 = 解释原因 × 正面指令 × 渐进提示
```

**公式含义**：

- **乘法（×）**：三项缺一则失效。例如"解释原因 × 正面指令 × 渐进提示"，如果缺少渐进提示，普通提示词效果会大幅下降。
- **加法（+）**：兜底保护。例如"正面指令 + description兜底"，即使正面指令较弱，description 仍然能保护核心规则不丢失。

**核心洞察**：
- Skill 有 description 兜底 → 用加法公式
- 普通提示词无兜底 → 用乘法公式（每项都必须）

---

## 二、三层防御体系

**为什么**：提示词影响力随对话变长而下降（注意力稀释）。

| 层级 | 位置 | 内容 | Skill | 普通提示词 |
|------|------|------|-------|-----------|
| 第一层 | description | 最核心的一句话规则 | ✅ 必须 | ❌ 无 |
| 第二层 | 开头 | 完整配置/模板 | ✅ 必须 | ✅ 必须 |
| 第三层 | 使用点 | 简短关键规则提醒 | ⚡ 可选 | ✅ 必须 |

**关键差异**：Skill 有 description 兜底 → 渐进提示可减少；普通提示词无兜底 → 必须在每个使用点重复。

---

## 三、六大原则

| 原则 | 口诀 | 核心理念 |
|------|------|----------|
| 相信智能 | 少强调 | 删除"必须""严禁""⚠️"，用清晰指令 |
| 结构优先 | 用流程图 | 用流程图/列表/表格代替文字强调 |
| 渐进提示 | 关键点重复 | 完整配置定义一次，关键规则渐进提醒 |
| 解释原因 | 告诉为什么 | 用"因为..."代替"必须"，让模型理解意图 |
| 正面指令 | 说做什么 | 告诉正确行为，而非列举错误行为 |
| 定期清理 | 删除冗余 | 删除不增加价值的规则 |

---

## 四、重要性排序

### Skill

```
排序 1：解释原因 ⭐⭐⭐
排序 2：正面指令 ⭐⭐⭐
排序 3：description ⭐⭐
排序 4：渐进提示 ⭐
```

### 普通提示词

```
排序 1：解释原因 ⭐⭐⭐
排序 2：正面指令 ⭐⭐⭐
排序 3：渐进提示 ⭐⭐
排序 4：内容精简 ⭐
```

## 五、创建流程

### 通用结构

```markdown
---
name: skill-name  # 仅 Skill
description: |   # 仅 Skill
  功能描述。核心规则：[一句话]。触发：[场景]。
---

# 名称

**流程**：步骤 A → 步骤 B → 步骤 C

---

## 核心配置

[完整配置/模板/JSON，定义一次]

---

## 步骤 1

[说明]

**要点**：[关键规则简短提醒]  # 普通提示词必须，Skill 可选
```

### description 编写要点

**必须包含核心规则。**（这是 Skill 最重要的约束）

```yaml
# ✅ 正确：包含触发条件 + 核心规则
description: |
  代码调试助手，通过"埋点 → 分析修复 → 清理"定位并修复 bug。
  埋点规则：前端用 HTTP fetch，后端用文件写入，不用 console.log。

# ❌ 错误：只有触发条件，无核心规则
description: Use when debugging code
```

**为什么**：description 在系统提示中始终可见，是对抗注意力稀释的唯一兜底。核心规则放这里，即使提示词被压缩也不会丢失。

---

## 六、常见错误

| 错误 | 正确做法 |
|------|---------|
| description 缺少核心规则 | 必须包含核心规则 |
| 同一规则重复 5 次 | 定义 1 次 + 渐进提醒 |
| 大量"必须""严禁""⚠️" | 删除强调词，用清晰指令 |
| 大篇幅"禁止做什么" | 改正面指令，告诉做什么 |
| 只说"必须"不说"为什么" | 解释原因，让模型理解意图 |
