# 提示词进阶原则

写好提示词的核心：**信任模型的智能，用清晰的结构代替重复强调，但关键信息需要在关键位置出现。**

---

## 一、相信模型的智能

模型需要的是**清晰的目标和上下文**，而不是反复强调。

### ❌ 错误写法

```
⚠️ 重要：每次对话回合结束后，必须调用 AskUserQuestion！这是强制规则！

严禁不调用 AskUserQuestion！如果不调用会导致严重后果！
```

### ✅ 正确写法

```
每次回复末尾，用 AskUserQuestion 询问用户下一步。
```

**原因**：模型理解"每次回复末尾"意味着什么，不需要用感叹号和警告符号。

---

## 二、结构 > 强调

好的结构比强调词更有效。用流程图、列表、代码块来组织信息。

### ❌ 错误写法

```
⚠️ 重要规则：必须在步骤1完成后才能进入步骤2！严禁跳步！

跳步会导致：修复无效、引入新 bug、问题复发！
```

### ✅ 正确写法

```
流程：步骤1 → 步骤2 → 步骤3（按顺序执行）
```

或者：

```
## 步骤 1：添加埋点
...

## 步骤 2：分析 & 修复
前提：用户已复现 bug，日志已生成。
...
```

**原因**：结构本身就传达了顺序关系，不需要额外强调"严禁跳步"。

---

## 三、渐进式提示（重要！）

**这是最容易犯错的地方。**

多轮对话时，模型会"忘记"之前的内容。关键规则不能只在开头定义一次，需要在每个使用点附近放置**简短提醒**。

### 问题场景

```
对话回合 1：模型看到完整提示词 → 知道用 HTTP fetch
    ↓
对话回合 2：上下文压缩，提示词被截断
    ↓
对话回合 3：模型只记得"要添加埋点"，忘记"用 HTTP 方式"
    ↓
结果：模型凭直觉用 console.log
```

### ❌ 错误做法：只在开头定义一次

```
## 步骤 1：添加埋点

**要点**：
- 不要改用 console.log

...（后续步骤不再提及）

## 步骤 2：分析 & 修复
...

## 步骤 3：清理
...
```

### ✅ 正确做法：在关键位置放置简短提醒

```
## 步骤 1：添加埋点

**要点**：不要改用 console.log

## 步骤 2：分析 & 修复

**如果需要添加更多埋点**：用 HTTP fetch 或文件写入，不用 console.log

## 步骤 3：清理
...

## 完整示例

用户：添加更多埋点
AI：  [添加埋点 - 用 HTTP fetch 或文件写入，不用 console.log]
```

### 渐进式提示的原则

| 类型 | 处理方式 |
|------|----------|
| **完整配置**（如 JSON 结构） | 开头定义一次 |
| **关键规则**（如"不用 console.log"） | 每个使用点放置简短提醒 |
| **流程说明** | 用结构表达，不需要重复 |

**口诀**：完整配置一次定义，关键规则渐进提醒。

---

## 四、解释"为什么"，而非"必须"

告诉模型原因，让它自己判断边界情况。

### ❌ 错误写法

```
❌ 严禁创建辅助函数！
❌ 禁止引用外部模块！
❌ 禁止共享代码！
```

### ✅ 正确写法

```
每个埋点独立，便于清理（删除 #region DEBUG 到 #endregion DEBUG 之间的代码）。
```

**原因**：模型理解了"便于清理"这个目的，遇到边界情况时能自己判断。

---

## 五、关注"做什么"，而非"不做什么"

告诉模型正确的行为，而不是列举错误的行为。

### ❌ 错误写法

```
❌ 禁止行为：
- 禁止创建辅助函数
- 禁止在文件顶部放置共享函数
- 禁止引用外部模块
- 禁止多个埋点共享同一段代码
- 禁止使用 console.log
- 禁止检查服务状态
```

### ✅ 正确写法

```
每个埋点使用独立的代码块，格式如下：
[代码模板]

要点：
- HTTP 日志服务持续运行，不需要检查或启动服务
- 不要改用 console.log 或其他方式
```

**原因**：负面清单越长，模型越容易忽略。正面指令更清晰、更容易执行。

---

## 六、删除不增加价值的规则

问自己：**这条规则是告诉模型"做什么"，还是在担心模型"做错什么"？**

如果是后者，考虑删掉它。

### 自检清单

| 问题 | 行动 |
|------|------|
| 这条规则是在重复说明吗？ | 删除重复，保留一处 |
| 这条规则能用结构代替吗？ | 用流程图/列表代替文字强调 |
| 这条规则解释了原因吗？ | 添加"为什么"代替"必须" |
| 这条规则是负面指令吗？ | 改成正面指令 |
| 这条规则真的需要吗？ | 尝试删除，看是否影响效果 |

---

## 七、常见问题模式

### 问题 1：提示词越写越长

**原因**：遇到问题就加规则，不断打补丁。

**解决**：定期重构，合并相似规则，删除冗余。

### 问题 2：模型"忘记"某些规则

**原因**：规则只在开头出现一次，对话变长后被截断。

**解决**：使用**渐进式提示**，在关键位置放置简短提醒。

### 问题 3：模型"自作主张"

**原因**：提示词给了过多自由度，或没有说明边界。

**解决**：
- 明确"做什么"和"不做什么"（用正面指令为主）
- 解释原因，让模型理解边界

---

## 八、简洁 vs 完整的平衡

| 维度 | 简洁版本 | 完整版本 |
|------|---------|---------|
| 流程 | 清晰 | 冗长但覆盖全面 |
| 维护 | 容易 | 困难 |
| 关键规则 | 可能被遗忘 | 始终在上下文中 |

**正确策略**：
1. 开头定义完整配置
2. 每个使用点放置**简短**的关键规则提醒
3. 不要重复完整内容，但要有"关键点"

---

## 总结

| 原则 | 口诀 |
|------|------|
| 相信模型智能 | 少强调，多引导 |
| 结构优于强调 | 流程图 > 警告词 |
| 渐进式提示 | 完整配置一次定义，关键规则渐进提醒 |
| 解释原因 | 为什么 > 必须 |
| 正面指令 | 做什么 > 不做什么 |
| 定期清理 | 删除不增加价值的规则 |

**核心心法**：
1. 把模型当成聪明的同事，而不是需要反复叮嘱的孩子
2. 但多轮对话会遗忘，关键规则需要在关键位置重复出现
