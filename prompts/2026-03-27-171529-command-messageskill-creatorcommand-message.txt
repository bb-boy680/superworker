  为什么会这样？

  这是典型的**"打补丁"式写作**：

  问题：模型忘记调用 options
    ↓
  解决：加一条"必须调用"的规则
    ↓
  问题：还是忘记
    ↓
  解决：加更多强调词（⚠️、必须、强制）
    ↓
  问题：对话变长后还是忘记
    ↓
  解决：在每个步骤结尾都重复一遍
    ↓
  结果：SKILL.md 变得臃肿、重复、难以阅读

  核心原因：用"堵"的方式解决问题，而不是找到根本原因。

  ---
  正确的思路

  1. 信任模型的理解能力

  模型是智能的，它需要的是清晰的目标和流程，而不是反复的强调。

  ❌ 错误：每次对话回合结束后必须调用
  AskUserQuestion！这是强制规则！不调用是错误的！

  ✅ 正确：每完成一个步骤后，用 AskUserQuestion 询问用户下一步。

  2. 用"为什么"代替"必须"

  ❌ 错误：严禁跳步！跳步后果：修复无效、引入新 bug！

  ✅ 正确：按顺序执行，因为埋点数据是分析的基础，分析结果是修复的依据。

  3. 减少重复，保持 DRY

  同样的配置不需要写 5 遍，只需要在开头定义一次，后面引用即可。

  4. 关注"做什么"，而非"不做什么"

  ❌ 错误：禁止创建辅助函数！禁止引用外部模块！禁止共享代码！

  ✅ 正确：每个埋点使用独立的代码块，格式如下...

  提示词进阶原则

❯ sed -i '/#region DEBUG/,/#endregion DEBUG/d' 文件名                           
这个命令是啥意思                                                                

● 这个命令用于删除埋点代码块：

  sed -i '/#region DEBUG/,/#endregion DEBUG/d' 文件名

  逐部分解析

  ┌─────────────────┬──────────────────────────────────┐
  │      部分       │               含义               │
  ├─────────────────┼──────────────────────────────────┤
  │ sed             │ 流编辑器，用于文本处理           │
  ├─────────────────┼──────────────────────────────────┤
  │ -i              │ 直接修改文件                     │
  ├─────────────────┼──────────────────────────────────┤
  │ /模式1/,/模式2/ │ 从匹配模式1的行，到匹配模式2的行 │
  ├─────────────────┼──────────────────────────────────┤
  │ d               │ delete，删除                     │
  └─────────────────┴──────────────────────────────────┘

  工作原理

  原始代码：
  ────────────────────────────
  function foo() {
    // #region DEBUG [sessionId: xxx, port: yyy]
    await import('fs').then(...)
    // #endregion DEBUG

    const result = bar();
    return result;
  }
  ────────────────────────────

  执行命令后：
  ────────────────────────────
  function foo() {


    const result = bar();
    return result;
  }
  ────────────────────────────

  删除了从 #region DEBUG 到 #endregion DEBUG 
  之间的所有行（包括这两个标记行本身）。

  注意

  删除后会留下空行。如果想同时清理空行，可以加一步：

  sed -i '/#region DEBUG/,/#endregion DEBUG/d' 文件名
  sed -i '/^$/d' 文件名  # 删除空行
